# Feature: 03 Runner Console Log Output

## REQUIREMENTS SPEC

## Goal

Update the in-repo core runner so it can print full per-scenario event logs to the console.

## Feature Interpretation

Runner Console Log Output extends the existing in-repo core runner so that, in addition to suite/check pass output, it can display the full human-readable event logs already produced by scenario execution for each scenario run.

## Behavior

- The runner prints human-readable per-scenario event logs to the console.
- Each scenario log section includes the scenario name.
- Scenario logs preserve the exact event order produced by the scenario runner.
- Console output remains deterministic across repeated runs of the same scenarios.
- Scenario output is useful for debugging movement, attack, damage, death, failure events, and verification results.
- The runner still reports suite-level success/failure and overall process status.

## Rules

- The runner must reuse existing scenario log generation rather than introducing a second scenario logging system.
- The runner must not change gameplay behavior to support console output.
- The runner must not duplicate scenario execution logic.
- The runner must continue to run entirely outside Unity.
- The runner must preserve existing non-zero exit code behavior for suite or host failure.
- Scenario log output must remain aligned with the existing deterministic scenario runner/event system behavior.

## Edge Cases

- Scenarios containing invalid actions must still print their failure-related log lines.
- A failing scenario verification must still show the scenario’s event log output.
- If multiple scenarios run, each scenario’s output must remain clearly separated.
- Repeated runs must not interleave or reorder logs differently between executions.
- If a suite fails before all scenarios complete, console output must still make it clear which scenario logs were printed and where failure occurred.

## Constraints

- No Unity dependencies may be introduced.
- No gameplay rules may be altered.
- No duplicate scenario logic may be created in the runner.
- Console output changes must remain compatible with the existing pure C# core, scenario runner, and in-repo runner architecture.

## Interactions

- Uses the existing core scenario runner.
- Uses the existing scenario event log generation.
- Extends the existing in-repo core runner console output behavior.
- Builds on the current combat/event verification flow without replacing it.

## Event Hooks

- The runner does not define new gameplay events.
- It prints the existing ordered scenario event logs already generated by scenario execution.
- Printed logs must include movement, attack, damage, death, and failure-related events when they occur.

## Logging Requirements

- Logs must be human-readable.
- Logs must include the scenario name.
- Logs must preserve event order exactly.
- Logs must remain deterministic across runs.
- Logs must be useful for debugging movement, attack, damage, death, failure events, and verification results.
- Logs must remain readable alongside the existing suite/check output.

## Acceptance Criteria

- [ ] The in-repo core runner prints full per-scenario event logs to the console.
- [ ] Each scenario log section includes the scenario name.
- [ ] Printed scenario logs preserve the exact event order generated by scenario execution.
- [ ] Movement, attack, damage, death, and failure events are visible in console output when they occur.
- [ ] Console output remains deterministic across repeated runs.
- [ ] Verification-related output remains visible alongside scenario logs.
- [ ] Runner exit-code behavior remains unchanged: success returns `0`, suite/host failure returns non-zero.
- [ ] No Unity dependencies are introduced.
- [ ] Gameplay behavior and scenario execution behavior remain unchanged.

## AMBIGUITIES

- It is not yet defined whether per-scenario logs should always print or be gated behind a future flag.
- It is not yet defined exactly where scenario log sections should appear relative to suite/check output.
- It is not yet defined whether scenario verification summaries should print before or after the full event log for each scenario.
- It is not yet defined whether the runner should print all scenarios only from the scenario suite or also expose direct scenario-only output in the future.
